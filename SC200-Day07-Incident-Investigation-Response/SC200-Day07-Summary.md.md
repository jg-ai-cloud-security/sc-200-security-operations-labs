\# SC-200 Day 07 — Incident Investigation + Response



\## Overview

Today I focused on the incident investigation and response workflow in Microsoft Sentinel. The objective was to review how incidents are accessed, analyzed, validated, and documented as part of a SOC process.



\## Lab Activities Completed

\- Opened the Microsoft Sentinel Incidents page

\- Confirmed the workspace initially had no live incidents

\- Created a manual test incident to simulate an investigation scenario

\- Reviewed incident details and status

\- Built the Day 07 investigation architecture diagram

\- Prepared KQL validation steps in Log Analytics

\- Documented business scenario, business risk, security goal, and investigation workflow

\- Captured screenshots as portfolio evidence



\## Tools Used

\- Microsoft Sentinel

\- Log Analytics

\- KQL

\- Azure Portal

\- GitHub

\- Azure DevOps

\- OneNote



\## Key Skills Demonstrated

\- Incident triage workflow

\- Microsoft Sentinel incident review

\- Manual incident creation

\- KQL validation planning

\- SOC documentation

\- Security architecture thinking



\## Outcome

This lab demonstrated the end-to-end incident investigation workflow in Microsoft Sentinel. Although no live incidents were initially available, a manual test incident was created to simulate the analyst process and document the investigation lifecycle in a structured way.



\## Lab Note

Because the incident was manually created and not generated by live analytics rules, alerts, entities, and investigation graph data may be limited in this lab.
