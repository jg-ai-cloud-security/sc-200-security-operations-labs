\# Day 07 Incident Findings



\## Incident Summary

A manual test incident was created in Microsoft Sentinel to simulate the Day 07 investigation scenario for BrightSpark Electrical Services. This allowed the incident review process to be practised in a controlled lab environment.



\## Investigation Context

At the start of the lab, no live incidents were present in the Sentinel incident queue. To continue the exercise, a manual incident was created so the incident workflow could still be documented and demonstrated.



\## Evidence Reviewed

\- Microsoft Sentinel Incidents page

\- Manual incident details

\- Incident status and severity

\- Architecture workflow diagram

\- KQL validation planning in Log Analytics



\## Key Findings

\- The Sentinel workspace was accessible and ready for incident review

\- No live incidents were available during the lab window

\- A manual test incident was successfully created

\- The investigation workflow was documented end-to-end

\- KQL validation was prepared to support evidence review



\## Triage Assessment

\- Incident type: Manual test incident

\- Severity: Medium

\- Status: New

\- Alerts linked: None

\- Entities linked: Limited or unavailable

\- Investigation graph: Limited due to manual incident creation



\## Recommended Response

\- Use the manual incident as a documented training scenario

\- Validate supporting telemetry with KQL in Log Analytics

\- Continue practising triage, documentation, and response workflow

\- Use real analytics-generated incidents in future labs where possible



\## Analyst Reflection

This lab reinforced the structure of the Sentinel investigation process. Even without live incidents, the workflow was still valuable because it demonstrated how an analyst would access, review, validate, and document a security case.
